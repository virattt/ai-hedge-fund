# 业务逻辑

## 1. portfolio_manager

位置：`src/agents/portfolio_manager.py`

### 作用

把上游分析结果和风险约束，转换成每个股票的最终交易动作：

- `buy`
- `sell`
- `short`
- `cover`
- `hold`

一句话理解：它是“最终下单决策器”。

### 输入

主要来自 `state["data"]`：

- `portfolio`：当前现金和持仓
- `analyst_signals`：各分析师 agent 的信号
- `tickers`：本次需要决策的股票列表

同时会读取 `risk_manager` 的输出，重点使用：

- `remaining_position_limit`
- `current_price`

### 核心流程

1. 读取组合、信号和股票列表。
2. 读取风控结果，计算每个 ticker 最多还能交易多少股。
3. 汇总多个分析师信号，压缩成更简洁的结构，减少提示词长度。
4. 先通过规则函数计算“允许动作集合”。
5. 再让 LLM 只在允许范围内选择最终动作、数量、置信度和简短理由。
6. 把结果写回状态，供下游继续使用。

### 关键点

LLM 虽然理论上有 5 个动作可选，但对每个 ticker，真正可选的是一个动态子集，不是随便选。

规则大致如下：

- `hold`：始终允许
- `buy`：有现金且价格有效时才允许
- `sell`：已有多头仓位时才允许
- `short`：满足数量和保证金约束时才允许
- `cover`：已有空头仓位时才允许

所以这里的设计重点不是“让 LLM 自由决策”，而是：

- 先算硬约束
- 再让 LLM 在约束内做选择

### 它解决的问题

- 把“多分析师意见 + 风险限制”统一成可执行指令
- 防止 LLM 给出越权交易动作
- 当没有交易空间时，自动退化为 `hold`

## 2. risk_manager

位置：`src/agents/risk_manager.py`

### 作用

把分析信号转成“可执行的风险上限”。

一句话理解：它决定“能不能下单，以及最多能下多少”。

### 输入

函数输入是 `state: AgentState`。

其中主要使用 `state["data"]` 里的内容：

- `portfolio`：当前现金和持仓
- `tickers`：本次要评估的股票
- `start_date`、`end_date`：价格数据区间

另外也会从 `state` 中读取 API key 来获取市场价格数据。

### 输出

函数会返回更新后的状态，重点包括两部分：

- `messages`：追加一条风险分析结果消息
- `data`：把风险分析写回 `state["data"]["analyst_signals"][agent_id]`

下游最关心的是每个 ticker 的这些字段：

- `remaining_position_limit`：还能开的最大仓位金额
- `current_price`：当前价格
- `volatility_metrics`：波动率指标
- `correlation_metrics`：与已有持仓的相关性
- `reasoning`：风险结果的解释

### 核心流程

1. 计算每个标的的波动率。
2. 计算它与现有持仓的相关性。
3. 结合组合净值、现金和风险调整因子，得到可用仓位上限。
4. 把结果交给 `portfolio_manager` 作为硬约束。

### 它解决的问题

它解决的是“下单前仓位风控”问题，避免系统因为单一看涨信号而过度下注。

核心逻辑可以概括为：

- 波动越大，可用仓位越小
- 与现有持仓相关性越高，可用仓位越小
- 还要同时满足组合净值和现金约束

### 一句话总结

`risk_manager` 负责算风险边界，`portfolio_manager` 负责在边界内做最终交易决策。
